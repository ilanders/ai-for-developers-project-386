# Интеграционные пользовательские сценарии

## Контекст

На этом этапе проверяется работа приложения на уровне пользовательских сценариев в реальном браузере.  
Тесты должны убедиться, что фронтенд и бэкенд работают вместе, а основной путь бронирования проходит от начала до конца.

Так как в проекте нет постоянной базы данных, основной e2e-сценарий начинается с админской части: владелец календаря создает тип события через UI, после чего гость бронирует слот для этого типа события.

Дефолтные данные могут использоваться для удобства локальной разработки, но интеграционные тесты не должны неявно зависеть от них. Перед запуском тестов состояние приложения должно быть воспроизводимым.

## Общие предположения

- В приложении нет регистрации и авторизации.
- Админская часть доступна как интерфейс владельца календаря.
- Гостевая часть доступна отдельно и позволяет выбрать тип события и слот.
- Тип события содержит:
  - идентификатор;
  - название;
  - описание;
  - длительность.
- Гость бронирует слот для выбранного типа события.
- Слоты доступны в пределах ближайших 14 дней.
- Один и тот же временной интервал нельзя забронировать повторно, даже если выбран другой тип события.
- Бэкенд использует in-memory-хранилище, поэтому для тестов важно контролировать начальное состояние.

## Сценарий 1. Админ создает тип события

### Цель

Проверить, что владелец календаря может создать тип события через административный интерфейс, а созданные данные сохраняются на бэкенде.

### Предусловия

- Приложение запущено.
- Бэкенд доступен фронтенду.
- Состояние приложения перед тестом очищено или приведено к известному состоянию.

### Шаги

1. Открыть админскую часть приложения.
2. Перейти на страницу управления типами событий.
3. Нажать кнопку создания нового типа события.
4. Заполнить форму:
   - идентификатор: `consultation`;
   - название: `Consultation`;
   - описание: `Test booking event`;
   - длительность: `30`.
5. Сохранить тип события.

### Ожидаемый результат

- Тип события успешно создается.
- Админ видит созданный тип события в списке.
- В списке отображаются название, описание и длительность.
- После обновления страницы созданный тип события остается доступным, пока приложение не перезапущено или состояние не очищено.

## Сценарий 2. Гость видит созданный тип события

### Цель

Проверить, что тип события, созданный через админский интерфейс, доступен гостю в гостевом интерфейсе.

### Предусловия

- Через админский интерфейс создан тип события `Consultation`.

### Шаги

1. Открыть гостевую страницу приложения.
2. Просмотреть список доступных типов событий.

### Ожидаемый результат

- Гость видит тип события `Consultation`.
- У типа события отображаются:
  - название;
  - описание;
  - длительность.
- Гость может выбрать этот тип события для бронирования.

## Сценарий 3. Гость выбирает тип события и видит свободные слоты

### Цель

Проверить, что после выбора типа события приложение показывает доступные слоты для бронирования.

### Предусловия

- В системе есть тип события `Consultation`.
- Для выбранного типа события есть хотя бы один свободный слот в ближайшие 14 дней.

### Шаги

1. Открыть гостевую страницу.
2. Выбрать тип события `Consultation`.
3. Просмотреть список свободных слотов, сгруппированных по датам.
4. Выбрать свободный слот.

### Ожидаемый результат

- Приложение показывает свободные слоты для выбранного типа события.
- Слоты находятся в пределах ближайших 14 дней.
- Пользователь может выбрать свободный слот.
- Занятые слоты не отображаются как доступные.

## Сценарий 4. Гость успешно создает бронирование

### Цель

Проверить основной пользовательский путь бронирования от выбора типа события до успешного подтверждения.

### Предусловия

- В системе есть тип события `Consultation`.
- Есть хотя бы один свободный слот для бронирования.

### Шаги

1. Открыть гостевую страницу.
2. Выбрать тип события `Consultation`.
3. Выбрать свободный слот.
4. Заполнить данные гостя, если форма бронирования их требует.
5. Подтвердить бронирование.

### Ожидаемый результат

- Бронирование успешно создается.
- Пользователь видит сообщение или экран успешного бронирования.
- В подтверждении отображаются основные данные бронирования:
  - тип события;
  - дата;
  - время;
  - длительность;
  - данные гостя, если они используются в интерфейсе.

## Сценарий 5. Созданное бронирование отображается в админской части

### Цель

Проверить, что бронирование, созданное гостем, отображается владельцу календаря.

### Предусловия

- Гость успешно создал бронирование для типа события `Consultation`.

### Шаги

1. Перейти в админскую часть приложения.
2. Открыть страницу предстоящих встреч или бронирований.
3. Найти созданное бронирование.

### Ожидаемый результат

- Новое бронирование отображается в списке предстоящих встреч.
- В списке видны основные данные:
  - тип события;
  - дата;
  - время;
  - длительность;
  - данные гостя, если они есть.

## Сценарий 6. Забронированный слот нельзя выбрать повторно

### Цель

Проверить бизнес-правило, запрещающее двойное бронирование одного и того же времени.

### Предусловия

- Гость уже создал бронирование на конкретный слот.

### Шаги

1. Открыть гостевую страницу.
2. Выбрать тип события `Consultation`.
3. Найти в списке слотов дату, на которую было создано бронирование.
4. Убедиться, что забронированный слот отсутствует среди доступных.
5. При возможности попытаться выбрать ранее забронированный слот.

### Ожидаемый результат

- Ранее забронированный слот не отображается как доступный.
- Если слот все же отображается из-за устаревшего состояния интерфейса, повторная отправка бронирования отклоняется бэкендом.
- Пользователь видит понятное сообщение об ошибке.
- Второе бронирование на то же время не создается.

## Сценарий 7. Нельзя забронировать слот вне доступного диапазона

### Цель

Проверить ограничение бронирования ближайшими 14 днями.

### Предусловия

- Приложение рассчитывает доступные слоты на основе текущей даты и таймзоны.

### Шаги

1. Открыть гостевую страницу.
2. Выбрать тип события `Consultation`.
3. Просмотреть список доступных дат со слотами.
4. Убедиться, что все даты находятся в пределах ближайших 14 дней.

### Ожидаемый результат

- Пользователь видит только даты в пределах ближайших 14 дней.
- Даты за пределами диапазона недоступны для выбора или не отображаются.
- Нельзя создать бронирование на дату за пределами разрешенного диапазона.

## Основной сквозной сценарий для первого Playwright-теста

Для первого e2e-покрытия рекомендуется сделать один длинный сквозной сценарий:

1. Админ открывает административную часть.
2. Админ создает тип события с идентификатором `consultation` и названием `Consultation`.
3. Админ видит созданный тип события в списке.
4. Гость открывает гостевую страницу.
5. Гость видит созданный тип события.
6. Гость выбирает тип события.
7. Гость выбирает свободный слот.
8. Гость подтверждает бронирование.
9. Гость видит сообщение об успешном бронировании.
10. Админ открывает список предстоящих встреч.
11. Админ видит созданное бронирование.
12. Гость снова открывает выбор слотов.
13. Ранее забронированный слот больше не доступен для выбора.

Этот сценарий проверяет ключевую связку:

- создание типа события в админке;
- сохранение данных на бэкенде;
- отображение данных в гостевом интерфейсе;
- создание бронирования;
- отображение бронирования в админке;
- защиту от повторного бронирования.

## Дополнительные сценарии после базового покрытия

После реализации основного сквозного сценария можно добавить отдельные тесты:

1. Валидация формы создания типа события.
2. Валидация формы бронирования.
3. Проверка пустого состояния, когда типов событий еще нет.
4. Проверка пустого состояния, когда для выбранного типа события нет свободных слотов.
5. Проверка ошибки бэкенда при попытке повторного бронирования.
6. Проверка ограничения доступного диапазона ближайшими 14 днями.

## Рекомендации для будущей реализации Playwright-тестов

- Не завязывать тесты на дефолтные данные приложения.
- Перед тестом приводить состояние приложения к известному виду.
- Основной e2e-сценарий выполнять через UI, а не через прямые API-вызовы.
- Для вспомогательных тестов можно использовать API-подготовку данных, если это упростит сценарий.
- Использовать стабильные пользовательские селекторы, например `data-testid`, только там, где роли и текстовые селекторы недостаточны.
- Проверять не внутреннюю реализацию, а видимое пользователю поведение.
- В CI запускать фронтенд, бэкенд и Playwright-тесты вместе.
